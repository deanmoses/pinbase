# Model-Driven Metadata — Violations

Hand-maintained registries / maps that duplicate information Django already knows, per the criteria in [ModelDrivenMetadata.md](ModelDrivenMetadata.md). This doc has two layers: the raw inventory (one row per active violation) and a grouping of those rows by consuming subsystem into analytical clusters.

Current count: **six active violations**. PR #281 removed the two claims-metadata field-on-model examples from the umbrella doc and consolidated three relationship-schema registries into one `RelationshipSchema` registry. The central relationship registry is still hand-maintained, but that shape is an explicit deferral decision in [ModelDrivenCatalogRelationshipMetadata.md](ModelDrivenCatalogRelationshipMetadata.md), not an active duplicate-registry cleanup item.

## Findings: four clusters

Grouping the six active violations by consuming subsystem produces four clusters. These are analytical groupings, not a delivery plan.

### Cluster 1 — Residual claim-relationship metadata (3 of 6)

`M2M_FIELDS`, `_parent_dispatch`, `_custom_dispatch`. These remain after the relationship-schema consolidation and answer the resolver side of the same question: _given a claim namespace, what materialized relationship does it affect, and which resolver should run?_

The old `_entity_ref_targets`, `_literal_schemas`, and `_relationship_target_registry` rows are gone. They were consolidated into `register_catalog_relationship_schemas()` / `_relationship_schemas`, which now drives claim construction, namespace enumeration, write-path shape validation, and batch relationship FK existence checks. Moving those declarations onto model classes is deferred until a second independent consumer appears.

Detailed design — the concrete spec, the `Theme.parents` / `GameplayFeature.parents` promotion prerequisite, and the open questions (M2M/bespoke asymmetry, multi-subject handling, resolver signatures, app boundary) — lives in [ModelDrivenCatalogRelationshipMetadata.md](ModelDrivenCatalogRelationshipMetadata.md).

### Cluster 2 — Citation source identity (1 of 6, plus overlap with cluster 4)

The `identifier_key` touch-points (extractor dict, Django enum, CHECK constraint, seed data). One axis, subsumed by `CitationSourceSpec`.

Detailed design — the consumers, the open ownership question, the migration-story caveat for the CHECK constraint, and the dependency on `CatalogRelationshipSpec` — lives in [ModelDrivenCitationSourceMetadata.md](ModelDrivenCitationSourceMetadata.md).

### Cluster 3 — Introspection-derivable lists (1 of 6)

`_SOURCE_FIELDS` in `citation/seeding.py`. This does not need a spec; it is a Shape 1 `_meta` walk one parity-test away from being derived.

The cache-invalidation signal list is no longer active inventory: `catalog/signals.py` now derives concrete `CatalogModel` subclasses from the app registry and appends the few explicit extras that are not top-level catalog entities.

### Cluster 4 — Ingest adapter wiring (1 of 6)

`resolve_hooks`. Ambiguous inclusion. Not a third axis; sits at the intersection of clusters 1 and 2 — adapters declare which relationships they write (touches `CatalogRelationshipSpec`) and they _are_ citation sources (touches `CitationSourceSpec`). Revisit once both specs exist.

## Notes on excluded candidates

- `LinkType` registry (`core/markdown_links.py`) — populated from model class attrs (`link_label`, `link_sort_order`, …) in `CatalogConfig.ready()`. Already the correct "class-attr + derivation" pattern.
- `_ENTITY_TYPE_MAP` (`core/entity_types.py`) — derived from `LinkableModel` subclasses. Listed in the parent doc as the canonical example.
- `ENTITY_TYPE_ALIASES` / `_ENTITY_MODEL_MAP` — no longer exist; both were removed. The `project_entity_type_alias_consolidation` memory note was retired.
- IPDB `CLAIM_FIELDS` / `CREDIT_FIELDS`, `IPDB_TAG_MAP`, `PROP_TO_ROLE`, `_LABEL_TO_ROLE`, `_COUNTRY_NORMALIZATION`, `_STATE_NORMALIZATION` — source-specific translation tables (external-name → internal-slug). Not duplicates of Django model metadata.
- `register_catalog_relationship_schemas()` / `_relationship_schemas` — deliberately accepted transitional shape. It replaced three duplicated registries (`_entity_ref_targets`, `_literal_schemas`, `_relationship_target_registry`) with one authoritative runtime registry. Moving that registry onto model classes is deferred until there is a second independent consumer.
- Cache-invalidation signal list — no longer hand-maintained; `catalog/signals.py` derives concrete `CatalogModel` subclasses and appends explicit non-entity extras.
- `MEDIA_CATEGORIES`, `claim_fk_lookups`, `claims_exempt`, `soft_delete_cascade_relations`, `soft_delete_usage_blockers`, `api_type_key` — already on model classes as the correct pattern.
- Manufacturer / corporate-entity lookup dicts in `bulk_utils.py` — in-function caches populated from DB queries, not hand-maintained metadata.
- Rate-limit constants, media dimension constants, licensing ranks — module-level configuration orthogonal to any specific model.
- Ninja router autodiscovery (`config/api.py`) — framework contract.
